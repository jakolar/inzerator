# Viewer / realtor-overlay extraction

**Date:** 2026-05-09
**Repo:** inzerator (newly spun off from gtaol)
**Scope:** Refactor `hnojice_multi.html` into a generic generated viewer
+ a separate `viewer-realtor-overlay.js` ES module that adds the
realtor-specific UI and behaviour. Update `gen_multitile.py` template
so future regenerations preserve realtor features.
**Goal:** `gen_multitile.py` can regenerate any `<location>_multi.html`
without losing realtor superpowers (drone video tool, parcel highlights,
single-parcel click-to-highlight, free-flythrough). Today, regeneration
wipes ~1500 lines of manual additions.

## Motivation

`hnojice_multi.html` is currently ~2500 lines. About 1000 of those are
generated-viewer scaffolding (terrain GLB loading, ortofoto texturing,
OrbitControls, basemap UI, building popup, cadastre raster) and ~1500
are realtor features bolted on after generation: parcels layer,
single-parcel click highlight (painted-on-mesh), drone video panel
with 9 cinematography presets and 6 toggleable highlights, free
flythrough, sunset tint. If the user runs `gen_multitile.py` again
to refresh tiles or change a parameter, all the realtor code is
overwritten by the template.

The fix is conventional — split into a generated base and a stable
overlay module loaded at runtime.

## Scope

**Mid extract** (chosen from three options):

- **Stays in generated `<location>_multi.html`:** terrain GLB loading,
  ortofoto texture pipeline, OrbitControls, basemap selector, height
  exaggeration, wireframe toggle, cadastre raster overlay, basic
  building popup (č.p. / plocha / Mapy.cz / RÚIAN detail links),
  `getTerrainHeightAt` helper, the rAF tick loop with hookable API.
- **Extracted to `viewer-realtor-overlay.js`:** parcels layer (1999-tile
  3D extrusions), single-parcel click-to-highlight (painted-on-mesh),
  drone video panel + 9 presets + 6 highlights + sunset tint + MP4
  export, free-flythrough button, building-popup `🎬 Vytvořit video`
  link injection, `_sceneOverlay` abstraction.

The split puts a natural boundary between "generic Czech-village 3D
viewer" (reusable for any non-realtor 3D-tour use case — hiking
preview, archaeology, urbanistics) and "realtor superpowers".

## Architecture

### Two-file layout

```
inzerator/
├── hnojice_multi.html          # generated by gen_multitile.py — generic viewer
├── viewer-realtor-overlay.js    # standalone ES module — realtor stack
└── gen_multitile.py             # template emits viewer with overlay <script> reference
```

Other location viewers (e.g. `strazek_multi.html`, `praha_multi.html`)
are gitignored and produced by `gen_multitile.py`. They all reference
the same `viewer-realtor-overlay.js` next to them.

### Base ↔ Overlay contract

Generated viewer's `<script type="module">`, after setup, executes:

```js
try {
  const overlay = await import('./viewer-realtor-overlay.js');
  overlay.init({
    THREE,
    scene, camera, renderer, controls,
    allMeshes,                // array of terrain GLB meshes (populated after dataLoaded)
    ruianBuildings,           // building footprints array (from <location>_data.json)
    gcx, gcy,                 // SJTSK origin of this location
    addTickHook,              // (fn) => fn called every frame after main tick
    removeTickHook,           // (fn) => unregister
    setMainTick,              // (fn) => override default interactiveTick (used by video preview/record)
    resetMainTick,            // () => restore interactiveTick
    getBuildingPopup,         // () => returns the existing #building-popup element
    getTerrainHeightAt,       // (x, z) => terrain Y via raycast against allMeshes
  });
} catch (e) {
  console.warn('realtor overlay not loaded:', e);
}
```

Overlay returns nothing — all state lives in its closure. The base
viewer never reaches inside.

If `viewer-realtor-overlay.js` is missing or fails to load, the base
viewer continues to work as a plain 3D village viewer (no realtor UI,
no errors that break the page).

### Tick hook API in base

The current base tick is:
```js
function interactiveTick() { controls.update(); renderer.render(scene, camera); }
let _currentTick = interactiveTick;
function tick() { _currentTick(); requestAnimationFrame(tick); }
tick();
```

Refactor to support overlay hooks:
```js
function interactiveTick() { controls.update(); renderer.render(scene, camera); }
let _mainTick = interactiveTick;
const _tickHooks = [];
function tick() {
  _mainTick();
  for (const hook of _tickHooks) hook();
  requestAnimationFrame(tick);
}
tick();

// Exposed to overlay via init:
const addTickHook    = fn => _tickHooks.push(fn);
const removeTickHook = fn => { const i = _tickHooks.indexOf(fn); if (i >= 0) _tickHooks.splice(i, 1); };
const setMainTick    = fn => { _mainTick = fn; };
const resetMainTick  = ()  => { _mainTick = interactiveTick; };
```

Overlay uses:
- `addTickHook(_sceneOverlay.tick.bind(_sceneOverlay))` once on init — animations always run.
- `setMainTick(videoTick)` when Preview/Export starts — camera path animation takes over rendering.
- `resetMainTick()` when animation completes / cancels — interactive control resumes.

This separates "always-on per-frame work" (highlights animation) from
"takeover renderer" (video animation), with no coupling to overlay
internals.

### Overlay (`viewer-realtor-overlay.js`)

Single ES module exporting one function:

```js
export function init(args) {
  // 1. Inject CSS into <head>
  document.head.insertAdjacentHTML('beforeend', `<style>${CSS}</style>`);

  // 2. Inject HTML into <body>
  document.body.insertAdjacentHTML('beforeend', VIDEO_PANEL_HTML);
  // …etc for #parcelsBtn, #freeRecordBtn, #video-mode-hint
  // (or insert into the existing #info panel at appropriate spot)

  // 3. Wire all event handlers using args.scene, args.camera, etc.
  // 4. Register tick hook for animations
  // 5. Patch building popup template (intercept left-click on building,
  //    inject `🎬 Vytvořit video` link into popup contents)
}
```

CSS and HTML are template literals at the top of the file (constants),
not separate files — single overlay file, no fetch races.

All `_video*`, `_hl*`, etc. state lives inside `init()`'s closure or
at module top-level (overlay-private).

### gen_multitile.py template

Inside the existing template emit (around line 760-770 currently),

```html
<script type="module">
  // …existing base viewer module code…

  // Try-load realtor overlay (file may or may not be present).

```

Move out of template (these go into overlay file, not into generated
HTML): all video/parcel/highlight code blocks.

## Data flow

**Page load:**
1. Browser parses generated viewer HTML
2. Module script runs: imports THREE, sets up scene
3. `dataLoaded` fetches `tiles_<location>/<location>_data.json`
4. Then GLB tiles loaded, ortofoto textures applied
5. UI handlers wired (basemap, wireframe, cadastre, building click)
6. `tick()` starts
7. Try-import overlay: if file present, `init({...})` runs; injects
   realtor UI; registers tick hook

**Realtor flow (after overlay loaded):**
1. User clicks "Parcely" → overlay's `ensureParcels` → `/api/parcels`
   fetch → `buildParcelGroup` → `scene.add(group)`
2. User left-clicks anywhere → base building-popup fires (if hit a
   building) → overlay's intercepted popup-video link present →
   user clicks 🎬 → overlay opens video panel
3. User right-clicks parcel → overlay's contextmenu handler →
   `_sceneOverlay.apply(subject, mode)` → presentation mode + highlights
4. User clicks Preview → overlay calls `setMainTick(videoTick)` →
   camera animates → on completion `resetMainTick()`

## Error handling

- **Overlay fails to load** (network 404, syntax error): try-catch
  swallows, base viewer works without realtor UI. `console.warn`
  surfaces the failure for diagnosis.
- **Overlay init throws** (e.g., expected DOM element missing in older
  base): same try-catch, base survives.
- **Server-side endpoints missing/error** (parcels API down): existing
  per-feature error handlers in overlay continue to work (toast in
  popup, console error, button text changed).
- **Old generated viewer (pre-refactor)** opening with NEW overlay
  loaded: the try-import attempt may succeed but `init({...})` will
  fail because base didn't pass the new args (addTickHook, etc.).
  Try-catch handles this — overlay just doesn't activate. User regenerates.

## What we are NOT doing (YAGNI)

- **No build step / bundler / transpiler.** Vanilla ES modules +
  importmap are baseline 2024+ browser APIs.
- **No CSS / HTML in separate files.** Inline as template literals
  in the overlay JS — single file, no fetch races.
- **No overlay versioning.** Path is hardcoded `./viewer-realtor-overlay.js`.
- **No automated tests.** Manual browser verification per project
  convention.
- **No refactor of `inspector.html`.** Different domain (per-building
  detail vs whole-village). Separate concern.
- **No backwards-compat for old generated viewers.** Try-catch
  gracefully fails when init signature mismatch.
- **No partial extract via flags.** `viewer-realtor-overlay.js` is
  all-or-nothing: it loads, initializes the full realtor stack, or it
  doesn't load.
- **No multi-overlay system.** One overlay file, one `init()` call.
  If a future use case needs multiple overlay variants (e.g.,
  archaeology overlay), that's a separate refactor.

## Risks

- **Refactor introduces regression.** ~1500 lines moving between two
  files; references between presentation/highlight/sunset functions are
  intricate (`_sceneOverlay` orchestration). Mitigation: keep base and
  overlay function-for-function equivalent — only change scope, not
  logic. Manual test plan above runs the full flow.
- **Closure capture vs module load order.** Overlay references
  `scene`, `camera`, etc. captured at `init()` call time. If something
  in the base later replaces those bindings, overlay sees stale
  references. Currently none do (all are `const` in base), but worth
  verifying.
- **`gen_multitile.py` template breakage.** Adding `addTickHook` /
  `setMainTick` API + try-import block must work for ALL locations
  (Strážek, Snežka, Praha, Hnojice, Valcovní), not just Hnojice.
  Test by regenerating one other location too if possible.
